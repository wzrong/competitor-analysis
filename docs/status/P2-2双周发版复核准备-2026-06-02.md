## 二、复核动作

每家竞品按同一顺序检查，避免遗漏：

1. 打开发版追踪 `README.md`，读取「核心监控页面」和「待截图页面」。
2. 对照上一份基线记录，检查页面入口、导航、功能名称、价格/会员、App 版本、公开动态是否变化。
3. 有明显变化时截图；无变化不强制截图，只在记录中标注「无显著变化」。
4. 将变化分为四级：🚨 战略级、⚠️ 重要级、📊 运营级、ℹ️ 一般。
5. 判断是否触发 L9：决策层简报、Battlecard、销售话术卡、教研参考卡、运营参考卡。
6. 写入 `竞品库/{竞品名}/发版追踪/YYYY-MM-DD-发版记录.md`。
7. 更新该竞品 `发版追踪/README.md` 的「发版记录」表和「最后检查/下次检查」。
8. 如变化影响档案事实，更新对应 `profile.md`，并标注 T 级别和来源链接。
9. 如变化影响系统证据闭环，更新本清单、证据闭环清单和系统状态看板。

## 三、判定口径

| 变化类型 | 等级 | 处理动作 |
|---|---|---|
| 新产品、新核心模块、核心工作流重构、明确战略转向 | 🚨 战略级 | 发版记录 + profile 更新 + L9 触发判断，必要时启动深度分析 |
| 功能迭代、重要页面改版、新合作、新价格/会员入口 | ⚠️ 重要级 | 发版记录 + 截图 + 相关档案字段更新 |
| 促销、内容运营、资源量增长、社区活动、轻量会员权益变化 | 📊 运营级 | 发版记录；涉及增长/变现时标记运营参考卡候选 |
| 常规内容更新、页面文案微调、无结构变化 | ℹ️ 一般 | 发版记录中简记；通常不更新 profile |
| 无可见变化 | 无显著变化 | 发版记录中保留结论，不截图 |

## 四、截图规则

| 场景 | 推荐方式 | 说明 |
|---|---|---|
| 公开网页，无需登录 | Codex In-app Browser | 适合复核公开首页、功能页、会员页 |
| 需要登录态或真实浏览器环境 | Codex Chrome Extension | 适合作业帮、App Store、复杂交互页面 |
| 批量公开页面基线 | Chrome headless | 适合标准视口截图，但不处理登录态 |
| 临时留证 | macOS `screencapture` | 只作排障或临时人工证据 |

命名沿用项目规则：`{竞品名}-{页面描述}-YYYYMMDD.png`，保存到 `竞品库/{竞品名}/screenshots/`。
