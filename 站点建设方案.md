# 竞品分析工作台 — 对外开放网站方案

## Context

现有竞品分析工作台是 Obsidian 知识库中的一个 Project，包含：

- 23 个竞品档案（profile.md），每个含结构化字段
- 四维度分析框架与报告模板
- 每周自动监测简报（时间序列）
- 深度分析报告 + 截图

目标：将上述内容自动转换为**全体同事可访问的网站**，无需脱敏，支持搜索、导航、标签索引，且内容更新后**自动部署**。

---

## 方案概述

**技术栈**：MkDocs + Material 主题 + Python 构建脚本

**核心思路**：

1. 现有 Markdown 文件**原地不动**，继续作为 Obsidian 知识库的编辑入口
2. 一个 Python 构建脚本（`scripts/build.py`）将 Obsidian 文件转换为 MkDocs 的 `docs/` 目录结构
3. MkDocs 构建生成静态网站
4. 部署到内网服务器 / GitHub Pages（二选一）
5. 文件变更自动触发重新构建和部署

**为什么不直接用 Docusaurus？**
MkDocs Material 对纯文档场景更轻量，中文搜索和导航体验更好，配置简单，不需要 Node.js 环境。

## 实施步骤

### Step 1：初始化 MkDocs 项目（1 小时）

在 `Projects/竞品分析工作台/` 下创建网站子目录：

```bash
cd Projects/竞品分析工作台
python3 -m venv venv
source venv/bin/activate
pip install mkdocs-material mkdocs-git-revision-date-localized-plugin
mkdocs new website
cd website
```

### Step 2：编写构建脚本（2 小时）

创建 `scripts/build.py`，功能：

1. 读取 Obsidian 源文件（`Projects/竞品分析工作台/` 下的所有 md/png）
2. 按 slug 映射规则复制到 `website/docs/` 目录
3. **重写图片引用路径**：Obsidian 的 `竞品库/希沃/screenshots/xxx.png` → MkDocs 的 `../assets/screenshots/seewo/xxx.png`
4. 注入 YAML front matter（用于标签、日期、搜索优化）
5. 生成 `mkdocs.yml` 导航配置

脚本核心逻辑已在 Plan agent 输出中提供，需根据实际路径微调。

### Step 3：配置 MkDocs（1 小时）

`website/mkdocs.yml` 关键配置：

- Material 主题，中文语言
- 搜索插件（支持中文分词）
- Git 修订日期插件（显示每页"最后更新"时间）
- Tags 插件（支持标签索引页）
- 导航结构：tabs + sections + expand

### Step 4：处理图片路径（0.5 小时）

现有 Markdown 中的图片引用示例：
```markdown
![希沃首页](竞品库/希沃/screenshots/希沃-首页-20260518.png)
```

构建脚本需要将此路径转换为相对于目标 Markdown 文件的有效路径。

### Step 5：构建并本地预览（0.5 小时）

```bash
cd website
python3 ../scripts/build.py  # 生成 docs/
mkdocs serve                  # 本地预览 http://127.0.0.1:8000
```

### Step 6：配置自动部署（1-2 小时）

**方案 A：GitHub Pages（推荐，零服务器成本）**

1. 在 GitHub 创建私有仓库（如 `your-org/competitor-analysis-site`）
2. 将 `website/` 目录初始化为 git repo 并推送
3. 配置 GitHub Actions（`.github/workflows/deploy.yml`）：
   - 触发条件：`push` 到 main 分支
   - 执行：pip install → build.py → mkdocs build → deploy-pages
4. 本地配置 `fswatch` 监听 Obsidian 文件变更：
   ```bash
   fswatch -o /Users/wzrong/Documents/Claude/Projects/竞品分析工作台/ | while read f; do
     cd /path/to/website && python3 scripts/build.py && git add . && git commit -m "sync: $(date)" && git push
   done
   ```

**方案 B：内网服务器（rsync）**

1. 内网部署 Nginx，配置静态站点目录
2. 本地执行 `mkdocs build`，输出到 `site/` 目录
3. `rsync -avz site/ user@internal-server:/var/www/competitor-analysis/`
4. 同样可用 `fswatch` 监听文件变更，变更后自动执行 build + rsync

### Step 7：自定义样式（可选，1 小时）

- `docs/stylesheets/extra.css`：竞品相关度标签颜色（🔴🟡🟢）、卡片样式
- `docs/javascripts/extra.js`：可选的交互增强

## 风险与应对

| 风险 | 应对 |
|------|------|
| 中文文件名在 URL 中编码混乱 | 使用 slug 映射，所有 URL 为英文 |
| 图片路径因目录结构调整失效 | 构建脚本统一重写，不依赖原始相对路径 |
| 自动部署产生大量无意义 commit | 配置 debounce（变更后等待 5 分钟再同步）或改为每日定时同步 |
| Obsidian 与网站内容双写冲突 | Obsidian 始终是唯一编辑入口，网站只读展示 |
| 竞品库新增/删除竞品后导航不同步 | 构建脚本每次都重新生成 `mkdocs.yml` 的 nav 配置 |
